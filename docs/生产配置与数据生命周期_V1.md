# 生产配置与数据生命周期 V1

> 适用：`api/storage.py` 已启用匿名命盘的 owner-secret、删除与 TTL 后的任何预发布/生产环境。  
> 目标：让“30 天删除、仅本人可读、不留原始出生请求”不只停留在本地代码，而成为可验证的部署条件。

## 1. 先说结论

没有满足本文件的运行环境，**不得**把 Anima Codex 接到公开域名、真实支付或真实用户数据。

当前后端的本地开发回退会在未配置 `ANIMA_OWNER_SECRET_PEPPER` 时使用进程随机密钥。它故意让服务重启后无法再验证旧匿名凭据，因此适合本地演示，**不适合生产**。生产必须由密钥管理系统提供稳定的 HMAC pepper。

## 2. 生产环境变量清单

不要把真实值写入仓库、截图、客服工单、浏览器或前端构建产物。

| 变量 | 生产要求 | 原因 |
|---|---|---|
| `ANIMA_DB_PATH` | 受限服务账户可读写的绝对路径；备份路径与运行库分离 | 避免把数据库放入静态站点、临时目录或代码仓库。 |
| `ANIMA_OWNER_SECRET_PEPPER` | 由密钥管理系统注入；至少 32 个随机字节；不可打印 | 用于 HMAC 匿名所有权密钥。丢失会使旧匿名密钥失效；泄露要求轮换并评估失效范围。 |
| `ANIMA_CHART_TTL_SECONDS` | 正整数；首发建议 `2592000`（30 天） | 匿名命盘到期后必须自动清除。禁止用 `0` 或负数表达永久保存。 |
| `ANIMA_ENV` | 仅允许受控的 `development` / `staging` / `production` 值 | 用于部署脚本区分本地回退与生产门禁。当前代码尚未读取它；上线前须补生产启动校验。 |
| `ANIMA_ALLOWED_ORIGINS` | 仅列实际 HTTPS 前端域名 | 当前代码仍锁定本地前端源；上线前须将 CORS 改为按受控配置读取，不能开放 `*`。 |

示例只表示变量名，不提供任何真实密钥：

```dotenv
ANIMA_ENV=production
ANIMA_DB_PATH=/var/lib/anima-codex/anima.db
ANIMA_OWNER_SECRET_PEPPER=INJECT_FROM_SECRET_MANAGER__DO_NOT_COMMIT
ANIMA_CHART_TTL_SECONDS=2592000
ANIMA_ALLOWED_ORIGINS=https://app.example.com
```

## 3. 数据生命周期（匿名 Origin / First Scroll）

1. 用户提交出生资料时，排盘引擎仅在本次计算内使用原始请求。
2. 数据库只保存派生的 `ChartResult`、owner-secret 的 HMAC 摘要、创建时间与到期时间；不得保存 `request_json`。
3. 服务只在创建响应中交付高熵 `owner_secret`；它不进入 URL、日志、数据库明文或之后的 GET 响应。
4. GET、`/full`、`/rhythm` 与 DELETE 均要求 `X-Anima-Chart-Owner`。缺失、错误、过期与不存在统一返回 404，避免枚举。
5. 用户调用 DELETE 后立刻不可再读取；系统在启动、读、写、删操作时清理到期记录。
6. 到期匿名记录不进入备份恢复白名单。若备份保留期长于 30 天，恢复流程必须在恢复后先执行到期清理。

付费账户的数据保留、导出与删除会在账户/订单上线时另立版本；不得用“付费”作为延长保存出生资料的默认理由。

## 4. 部署前不可跳过的检查

### 应用与网络

- HTTPS 终止于受维护的反向代理；HTTP 一律跳转 HTTPS。
- API 与网页同域或 CORS 只允许明确的 HTTPS 站点。不能使用 `*`。
- 反向代理与应用均设置请求体大小、超时、按 IP / 用户账户的限流；当前仓库尚未实现生产级分布式限流，公开上线前必须补齐。
- 禁止在 URL、访问日志、分析系统、Sentry 类错误追踪中记录 owner secret、出生日期、时间、经纬度、城市或完整请求体。
- 数据库文件、备份和运行日志只能由最小权限服务账户访问；不放进 web root。

### 最小验证脚本

每次部署后必须用非真实出生资料验证：

1. POST `/api/v1/chart` 返回 `access.provider=owner_secret` 和到期时间；
2. 无 owner header 的 GET、`/full`、`/rhythm`、DELETE 均为统一 404；
3. 带正确 header 的 GET 可读，但响应中不含 `access`；
4. 正确 DELETE 返回 204，随后 GET 为 404；
5. 检查 `PRAGMA table_info(charts)` 不含 `request_json`；
6. 重启后仍可用已有 owner secret 读取（证明生产 pepper 稳定）；
7. 运行 `pytest tests/test_api.py -q` 的隔离数据库测试。

## 5. 监控、备份与事件响应

### 允许记录的最小指标

- 请求路径类别、状态码、延迟、匿名的聚合错误码、删除次数、到期清理数量；
- 不记录 chart ID、owner secret 或任何出生数据。若需要关联一次事故，使用短期、不可反推个人身份的请求追踪 ID。

### 备份

- 数据库备份加密；密钥与备份分离保存；恢复演练至少每季度一次。
- 恢复演练包括：过期记录被清理、已删除记录不因恢复重新暴露、无 `request_json` 列。
- 备份保留期、物理清除时限与用户隐私政策必须一致并可对外解释。

### owner-secret 疑似泄露

1. 立即撤销受影响记录或使其失效，不通过“继续可读”换取便利；
2. 轮换 `ANIMA_OWNER_SECRET_PEPPER` 前先评估：轮换会使现有匿名密钥全部失效；
3. 记录事件、影响范围、恢复措施与需要通知的用户；
4. 修补日志、前端存储或第三方脚本的泄露路径后再恢复服务。

## 6. 与真实收款的关系

当前匿名 owner secret 只解决“当前会话中谁可以读取/删除匿名命盘”，**不是账户体系，也不是支付授权**。

在打开真实 checkout 前，还必须完成：Magic Link 或等效账户所有权、服务端订单/权益状态机、支付 webhook 验证、退款/税费/取消政策、付费报告导出删除、生产级限流与监控。前端不得把 localStorage、支付成功页或 URL 参数当成 Full Codex 的授权。
